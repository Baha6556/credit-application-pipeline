# Пайплайн обработки кредитных заявок

Тестовое задание. Идея простая: REST API принимает заявку на кредит, проверяет её,
кладёт в очередь, а отдельный воркер решает — одобрить или отказать — и пишет
результат в базу.

Путь заявки:

```
клиент -> FastAPI (валидация) -> RabbitMQ -> воркер (скоринг) -> PostgreSQL
```

Если заявка невалидная, она дальше API не уходит — клиент сразу получает 422 со
списком ошибок. Валидная получает свой UUID, встаёт в очередь, и клиент может
позже спросить результат по `GET /applications/{id}`.

## Запуск

Нужен только Docker:

```bash
docker compose up --build
```

Поднимется четыре контейнера:

- `api` — само API, http://localhost:8000 (Swagger: http://localhost:8000/docs)
- `worker` — обработчик очереди
- `rabbitmq` — очередь, панель управления на http://localhost:15672 (guest/guest)
- `postgres` — база с результатами

Порт постгреса наружу я специально не пробрасывал — API и воркер ходят к нему по
внутренней сети compose, а на хосте 5432 частенько занят локальным постгресом.

Остановить: `docker compose down`.

## Как пощупать

Проще всего через Swagger (http://localhost:8000/docs): открываете
`POST /applications`, жмёте Try it out и отправляете что-то такое:

```json
{
  "client_id": 2001,
  "full_name": "Далер Назаров",
  "birth_date": "1990-05-10",
  "national_id": "12AB3456789",
  "phone": "+992901234567",
  "email": "daler@example.com",
  "monthly_income": 10000,
  "employment_status": "employed",
  "employment_duration_months": 36,
  "requested_amount": 24000,
  "requested_term_months": 24,
  "existing_loans_count": 0,
  "region": "Душанбе",
  "marital_status": "married",
  "dependents_count": 1,
  "application_date": "2026-07-10"
}
```

В ответ придёт 202 и `application_id`. Подставляете его в
`GET /applications/{application_id}` — и видите решение, скор и из чего он
сложился. Эту заявку, кстати, одобрит: платёж 1000 при доходе 10000.

А если в той же заявке поставить `monthly_income: 2000` и
`requested_term_months: 6` — будет отказ: платёж 4000 при доходе 2000, PTI
сильно больше половины.

Заглянуть в базу напрямую:

```bash
docker compose exec postgres psql -U credit -d credit \
  -c "SELECT application_id, decision, score, pti FROM application_results;"
```

## Тесты

Тесты юнитовые, докер для них не нужен:

```bash
python -m venv .venv
.venv\Scripts\activate        
pip install -r requirements.txt
pytest -v
```

Покрыта валидация (битые телефоны, email, даты, отрицательные суммы, левые
статусы занятости и т.д.) и логика решения, включая граничные случаи — PTI ровно
на пороге, скор ровно на пороге одобрения.

## Структура

```
app/
  main.py            # FastAPI: приём заявки, выдача результата, /health
  schemas.py         # модель заявки и вся валидация
  scoring.py         # правило решения (чистая функция)
  worker.py          # консьюмер: валидация -> скоринг -> запись в БД
  queue.py           # работа с RabbitMQ, объявление очередей
  db.py              # таблица результатов, подключение к постгресу
  config.py          # настройки из env
  logging_config.py  # логи
tests/
  test_validation.py
  test_scoring.py
```

## Решения и почему так

**Валидация на входе в API.** Невалидную заявку нет смысла тащить через брокер —
отправитель должен сразу узнать, что у него не так, поэтому 422 с перечнем ошибок
по полям. При этом воркер валидирует заявку ещё раз: очередь — внешняя система,
и слепо доверять тому, что оттуда пришло, не хочется.

Отдельный вопрос был — что чистить, а что отклонять. Решил так: нормализую только
то, что восстанавливается однозначно (лишние пробелы, регистр — `"MARRIED"` или
телефон с пробелами принимаются), а всё, где пришлось бы гадать (email без
собаки, обрезанный номер), — отклоняю. В кредитной теме молчаливо «додумать»
данные хуже, чем отказать.

Сами правила: телефон строго `+992XXXXXXXXX`, даты только ISO, доход и сумма
положительные с разумными потолками, срок — целое от 3 до 60 месяцев, статусы —
строгие enum, возраст на дату заявки 18–75, дата заявки не в будущем.

**Очередь.** Очередь durable, сообщения persistent — рестарт брокера заявки
переживают. Для битых сообщений заведён DLQ (`credit_applications.dlq`): пусть
лежат там для разбора, а не гоняют по кругу в redelivery. Если RabbitMQ лежит,
API отвечает 503 — клиент хотя бы знает, что заявка не принята, и может
повторить.

**Воркер.** `prefetch_count=1` и ack только после коммита в базу: если воркер
умрёт посреди обработки, заявка вернётся в очередь. Из этого следует
at-least-once, поэтому запись сделана идемпотентной по `application_id` — повторная
доставка дублей не плодит. Ошибки развёл по типам: мусорное сообщение — в DLQ,
временная проблема (например, база недоступна) — nack с requeue и небольшой
паузой.

**Скоринг.** Держу его чистой функцией без I/O: легко тестировать и, если
захочется, подменить на модель. Правило двухступенчатое: сначала жёсткий стоп —
PTI (платёж к доходу) больше 0.5 — отказ без разговоров. Дальше баллы: до 40 за
PTI, до 30 за статус занятости, 10 за стаж от года, до 15 за отсутствие других
кредитов, 5 за небольшое число иждивенцев. Набрал 60 из 100 — одобрено. Каждый
шаг расчёта пишется в базу в поле `reasons` — по любому решению видно, откуда оно
взялось. Для кредитов, мне кажется, объяснимость важнее хитрости модели.

**База.** Одна таблица `application_results`: решение, скор, PTI, причины и
полный исходный payload (JSON) — этого хватает и для аудита, и для отладки.
Схему создаёт воркер на старте через `create_all` с ретраями (постгрес в компоузе
поднимается не мгновенно). Для продакшена, понятно, тут были бы миграции.

## Что осталось за скобками

Что бы я добавил, если бы это был не тестовый проект: Alembic вместо
`create_all`, авторизацию на API, метрики, correlation id в логах, чтобы заявку
можно было проследить по всем сервисам, retry с экспоненциальным бэкоффом вместо
простого requeue, и интеграционные тесты на testcontainers.
