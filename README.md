# Пайплайн обработки кредитных заявок

Тестовое задание: сервис принимает кредитную заявку через REST API, валидирует её,
кладёт в очередь, а отдельный воркер принимает решение (одобрить / отказать)
и сохраняет результат в базу данных.

## Как это работает

```
        POST /applications (JSON)
клиент ──────────────────────────> FastAPI (api)
                                     │
                     невалидная ◄────┤ валидация Pydantic
                     422 + причины   │
                                     ▼ валидная
                                  RabbitMQ (очередь credit_applications)
                                     │
                                     ▼
                                  воркер (worker)
                                     │  скоринг: одобрить / отказать
                                     ▼
                                  PostgreSQL (таблица application_results)
                                     ▲
клиент <─────────────────────────────┘
        GET /applications/{id} — решение, скор, причины
```

Путь заявки по шагам:

1. Клиент отправляет JSON с данными заявки на `POST /applications`.
2. API валидирует её (обязательные поля, типы, диапазоны, форматы).
   Невалидная заявка сразу получает `422` со списком ошибок и в очередь не попадает.
3. Валидная заявка получает `application_id` (UUID), уходит в RabbitMQ,
   клиенту возвращается `202 {"application_id": ..., "status": "queued"}`.
4. Воркер забирает заявку из очереди, повторно валидирует, применяет правило
   скоринга и сохраняет решение с причинами в PostgreSQL.
5. Клиент узнаёт результат: `GET /applications/{application_id}`.

## Как запустить

Нужен только Docker (Desktop). Из корня проекта:

```bash
docker compose up --build
```

Поднимутся 4 контейнера:

| Контейнер  | Что делает | Доступ снаружи |
|---|---|---|
| `api`      | FastAPI, приём заявок | http://localhost:8000 |
| `worker`   | обработка очереди, скоринг, запись в БД | — |
| `rabbitmq` | очередь сообщений | management UI: http://localhost:15672 (guest/guest) |
| `postgres` | база с результатами | только из сети compose (порт наружу не проброшен) |

Остановить: `docker compose down` (с очисткой данных: `docker compose down -v`).

## Как проверить руками

Самый простой способ — Swagger UI: открыть **http://localhost:8000/docs**,
развернуть `POST /applications` → «Try it out» → отправить пример ниже.

То же через curl:

```bash
curl -X POST http://localhost:8000/applications -H "Content-Type: application/json" -d '{
  "client_id": 1086, "full_name": "Sharif Karimov", "birth_date": "1980-03-24",
  "national_id": "49CC7681090", "phone": "+992988942330", "email": "user359@gmail.com",
  "monthly_income": 9335.54, "employment_status": "self_employed",
  "employment_duration_months": 74, "requested_amount": 30487.01,
  "requested_term_months": 6, "existing_loans_count": 2, "region": "Istaravshan",
  "marital_status": "divorced", "dependents_count": 2, "application_date": "2026-04-22"
}'
# -> 202 {"application_id": "...", "status": "queued"}
```

Через секунду результат уже в базе:

```bash
curl http://localhost:8000/applications/<application_id из ответа выше>
# -> {"decision": "rejected", "score": 0, "pti": 0.5443,
#     "reasons": ["monthly_payment=5081.17", "pti=0.5443", "hard reject: pti > 0.5"], ...}
```

(Этой заявке отказано честно: платёж 5081 сомони в месяц при доходе 9335 —
больше половины дохода.)

Невалидная заявка в очередь не попадает:

```bash
curl -i -X POST http://localhost:8000/applications \
  -H "Content-Type: application/json" -d '{"client_id": 1, "phone": "983590502"}'
# -> 422, в detail перечислены все ошибки по полям
```

Примечание для Windows: если в curl из Git Bash / PowerShell передаёте кириллицу
inline — терминал может испортить кодировку. Надёжнее Swagger UI или
`curl --data-binary @file.json` с файлом в UTF-8.

## Тесты

Тесты покрывают валидацию и логику решения, это чистые unit-тесты —
Docker и внешние сервисы не нужны:

```bash
python -m venv .venv
.venv\Scripts\activate          # Windows (Linux/macOS: source .venv/bin/activate)
pip install -r requirements.txt
pytest -v
```

## Структура проекта

```
app/
  main.py            # REST API: POST /applications, GET /applications/{id}, /health
  schemas.py         # модель заявки + все правила валидации (Pydantic)
  scoring.py         # правило принятия решения — чистая функция без I/O
  worker.py          # консьюмер RabbitMQ: валидация -> скоринг -> запись в БД
  queue.py           # публикация в RabbitMQ, объявление очередей (+DLQ)
  db.py              # таблица результатов и подключение к PostgreSQL (SQLAlchemy)
  config.py          # настройки из переменных окружения
  logging_config.py  # единая настройка логов для api и воркера
tests/
  test_validation.py # тесты валидации (битые телефоны/email/даты/диапазоны/enum)
  test_scoring.py    # тесты решения (одобрение, жёсткий отказ, граничные случаи)
Dockerfile           # общий образ для api и воркера
docker-compose.yml   # запуск всего окружения одной командой
requirements.txt     # зависимости
```

## Принятые решения и почему

### 1. Валидация на входе в API, а не в воркере

Невалидная заявка отклоняется сразу с `422` и списком причин. Логика: отправитель
должен немедленно узнать, что исправить, а гонять заведомо битые данные через
брокер бессмысленно. Воркер при этом валидирует заявку **повторно** — он не
доверяет содержимому очереди (туда теоретически может писать кто угодно).

Основные правила ([app/schemas.py](app/schemas.py)):

- обязательные поля и типы — за счёт строгой Pydantic-модели;
- форматы: телефон `+992XXXXXXXXX`, email, `national_id` вида `49CC7681090`,
  даты только ISO `YYYY-MM-DD`;
- диапазоны: доход `0 < x ≤ 500 000`, сумма `0 < x ≤ 1 000 000`,
  срок — целое `3..60` мес., стаж/кредиты/иждивенцы — неотрицательные целые;
- категории: `employment_status`, `marital_status` — строгие enum;
- согласованность: возраст 18–75 лет на дату заявки, дата заявки не в будущем.

Граница «почистить или отклонить»: нормализуем только то, что восстанавливается
однозначно — пробелы и регистр (`"MARRIED"`, `"+992 96 226 9966"` принимаются).
Всё, где пришлось бы гадать (email без `@`, обрезанный телефон), отклоняем:
в кредитном пайплайне молчаливое «додумывание» данных опаснее отказа.

### 2. Очередь: durable + DLQ

- Очередь и сообщения персистентны — заявки переживают рестарт брокера.
- Для «ядовитых» сообщений (битый JSON и т.п.) настроен dead-letter queue
  `credit_applications.dlq`: они уходят туда для разбора, а не крутятся
  в бесконечном redelivery.
- Если брокер недоступен, API отвечает `503` — клиент знает, что заявка
  **не принята**, и может повторить.

### 3. Воркер: at-least-once без дублей

- `prefetch_count=1` и ручной ack **после** записи в БД: если воркер упал посреди
  обработки, сообщение вернётся в очередь и не потеряется.
- Повторная доставка не создаёт дублей — запись по `application_id` идемпотентна.
- Ошибки разделены по типу: битое сообщение → в DLQ; временная проблема
  (БД недоступна) → requeue с паузой.

### 4. Скоринг — чистая интерпретируемая функция

[app/scoring.py](app/scoring.py) не знает ни про очередь, ни про БД — тестируется
в изоляции и при желании заменяется на ML-модель. Правило простое:

1. **Жёсткий отказ**: PTI = (сумма / срок) / доход > 0.5 — платёж съедает больше
   половины дохода.
2. Иначе баллы 0–100: PTI (до 40), статус занятости (до 30), стаж ≥ 12 мес (+10),
   текущие кредиты (до 15), иждивенцы ≤ 2 (+5). Порог одобрения — **60**.

Каждый шаг расчёта пишется в БД в поле `reasons` — по любому решению видно,
из чего оно сложилось (для кредитов это важно).

### 5. БД: одна таблица с полным контекстом решения

`application_results`: решение, скор, PTI, причины (JSON) и полный исходный
payload (JSON) — достаточно для аудита и отладки. Схема создаётся воркером при
старте (`create_all` с ретраями на случай, если Postgres ещё поднимается).

### Что бы добавил для продакшена

Alembic-миграции вместо `create_all`, аутентификацию API, метрики (Prometheus),
correlation id в логах, retry с экспоненциальной задержкой вместо простого
requeue, интеграционные тесты с testcontainers.
